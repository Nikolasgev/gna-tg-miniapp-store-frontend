Ниже — сводное, целостное ТЗ по фронтенду (с учётом того, что **и Mini App, и Admin Panel разрабатываются на Flutter**, таргет — Web).

---

## 1. Общие положения

Фронтенд-проекты:

1. **Telegram Mini App** — клиентское приложение для конечных пользователей (покупателей) в контейнере Telegram WebApp.
2. **Admin Web Panel** — административная панель для владельца бизнеса и персонала (доступ из браузера, Flutter Web).

Оба фронтенда могут быть реализованы:

* как **два отдельных Flutter-проекта**,
  либо
* как **монорепозиторий** с общими Dart-пакетами (общие модели, DTO, API-клиент, design system).

Минимальное требование — логическая изоляция: разные entrypoint’ы, разные environment-конфиги.

---

## 2. Технологический стек (Frontend)

**Базовый стек:**

* **Flutter (последняя LTS-совместимая версия)**

  * Цель сборки: **Web** (для Telegram Mini App и для Admin Panel).
* **State management:**

  * `flutter_bloc` (BLoC)
  * Опционально: **Octopus** (если в команде уже используется / есть экспертиза).
* **HTTP-клиент:**

  * `dio` (предпочтительно) с interceptors
  * Допустим `http`, но с обёрткой и интерсепторами.
* **Локальное хранилище:**

  * `hive` — для корзины и кеша каталогов.
  * `shared_preferences` — для лёгких настроек (язык, флаги онбординга и т.п.).
* **Телеграм WebApp SDK:**

  * Встраивание JS SDK Telegram через `dart:js`/`js` или готовые обёртки.
* **i18n:**

  * `flutter_localizations` + `intl`.
* **UI-библиотеки:**

  * Допустимы легковесные UI-пакеты, но без тяжёлых зависимостей для Web.
* **Тестирование:**

  * `flutter_test`, `bloc_test`, `integration_test`.

## 3. Общие архитектурные требования

### 3.1. Слоистая архитектура

Рекомендованная структура:

* **presentation/**

  * экраны, виджеты, BLoC-и; только UI- и state-логика;
* **application/**

  * use-case / service-слой (опционально);
* **domain/**

  * сущности (Entity), интерфейсы репозиториев;
* **data/**

  * DTO, репозитории, доступ к API, маппинг из/в domain;
* **core/**

  * общие утилиты, error handling, theme, routing, i18n, constants.

### 3.2. Взаимодействие с backend

* Один централизованный **API-клиент** (на базе `dio`) с:

  * базовым URL (зависит от окружения: dev/stage/prod);
  * интерсептором для:

    * добавления Auth-заголовка (Bearer token);
    * логирования запросов/ответов (минимально — в dev);
    * обработки 401/403 (автоматический logout/refresh).
  * унифицированной обработкой ошибок:

    * таймаут, сеть, валидационные ошибки, внутренние ошибки сервера;
    * возврат человеко-понятных сообщений для UI.

* Для Telegram Mini App:

  * при старте передавать **`init_data` и `business_slug`** в backend через `/telegram/validate_init_data`;
  * backend в ответ возвращает:

    * валидированный `telegram_user` (id, name, phone, locale, avatar_url, etc.);
    * бизнес-конфиг (цвета, логотип, настройки оплаты и др.);
    * JWT/токен для дальнейших запросов (опционально).

## 4. Telegram Mini App (Flutter Web)

### 4.1. Общие требования

* Работать **внутри Telegram WebApp контейнера**:

  * поддержка методов Telegram WebApp API (themeParams, viewport, HapticFeedback, MainButton и т.п. при необходимости).
* Оптимизация Web bundle:

  * включён `--release` и tree shaking;
  * минимизация подключённых пакетов;
  * lazy-loading (где возможно) крупных модулей.
* Поддержка мобильных браузеров (минимум — последние версии Chrome/Firefox/Safari на Android/iOS).
* Соответствие UX паттернам мобайл-приложений (bottom sheets, плавные анимации, адаптивные списки).

### 4.2. Структура экранов Mini App

#### 4.2.1. Экран инициализации / Onboarding

**Цель:**
Проверка `init_data`, загрузка конфигурации, авторизация пользователя.

**Функционал:**

* Получение от Telegram параметров запуска (через JS bridge).
* Вызов backend `POST /telegram/validate_init_data` с `init_data` и `business_slug`.
* Отображение:

  * skeleton-загрузки или splash;
  * ошибки, если валидация не прошла (с возможностью попробовать ещё раз).
* После успешной валидации:

  * сохранение `user` и `config` в `AuthBloc/ConfigBloc` либо единый `AppBloc`;
  * переход на **главный экран — Каталог**.

#### 4.2.2. Главный экран — Каталог

**UI:**

* Верхняя часть:

  * логотип/название бизнеса;
  * горизонтальный список категорий (scrollable chips / tabs).
* Основная область:

  * список карточек продуктов: фото, название, краткое описание (опционально), цена, кнопка `Добавить`.
* Поиск/фильтр:

  * поле поиска по названию/описанию;
  * фильтры по категории (связаны с горизонтальным списком категорий).

**Функционал:**

* Загрузка категорий и продуктов:

  * GET `/categories`, GET `/products?category=...&search=...` (точные эндпоинты — как в backend ТЗ).
* Поддержка пагинации либо lazy-loading (scroll).
* Добавление товара в корзину:

  * через `CartBloc` — запись в локальное хранилище (hive) + отображение изменения иконки корзины.
* Обработка ошибок:

  * отображение сообщения + кнопка retry при проблемах с сетью.

#### 4.2.3. Карточка товара (детальный экран)

**UI:**

* Крупное фото товара (галерея, если есть несколько изображений).
* Название, цена, краткое и подробное описание.
* Selector количества (Stepper).
* Кнопка `Добавить в корзину` / `Обновить количество`.

**Функционал:**

* Загрузка подробной информации о товаре (GET `/products/{id}`).
* Управление количеством через `CartBloc`.
* При отсутствии товара (не найден / архивирован) — статус и возвращение назад.

#### 4.2.4. Корзина

**UI:**

* Список позиций:

  * название товара, цена, количество, итог по позиции;
  * кнопки `+ / -` для изменения количества;
  * кнопка удаления позиции.
* Итоговый блок:

  * сумма по корзине;
  * возможные скидки/акции (если backend поддерживает);
  * кнопка `Оформить заказ`.

**Функционал:**

* Работа с локальным хранилищем (hive):

  * чтение текущей корзины;
  * обновление позиций и синхронизация с `CartBloc`.
* Обработка сценариев:

  * товар недоступен (проверка при переходе к оформлению, при необходимости — запрос к API).
* При нажатии `Оформить заказ` → переход на **Checkout**.

#### 4.2.5. Checkout / Оформление заказа

**Форма:**

* Имя (предзаполняется из Telegram, при наличии).
* Телефон (предзаполняется из Telegram, при наличии).
* Способ получения:

  * Самовывоз / Доставка (зависит от конфигурации бизнеса).
* Адрес (обязателен при доставке).
* Способ оплаты (из backend-конфига):

  * Наличные;
  * Онлайн (через провайдера).

**Функционал:**

* Валидация формы (email/телефон при необходимости, обязательность полей).
* Создание заказа:

  * `POST /orders`
  * в теле: список позиций, контакты, адрес, метод оплаты, источник (Telegram).
* Если оплата наличными:

  * отображение экрана `Заказ создан` со статусом `awaiting_payment` или `new`.
* Если онлайн-оплата:

  * backend возвращает `checkout_url` (и, возможно, доп. данные).
  * Mini App:

    * открывает `checkout_url` через Telegram WebApp API (`openInvoice`/`openLink` — согласно выбранному провайдеру) или через `openExternalLink`.
  * После возвращения:

    * фронт выполняет **polling** `GET /orders/{id}` с интервалом (например, 3–5 сек, ограниченное число попыток);
    * при статусе `paid` → отображение успешного экрана;
    * при `failed` → отображение ошибки и предложений (повторить оплату или сменить способ).

#### 4.2.6. История заказов

**UI:**

* Список заказов (последние N, пагинация при необходимости):

  * дата/время;
  * сумма;
  * статус (цветовые метки: new, accepted, preparing, ready, completed, canceled, paid/unpaid).
* Детальный экран заказа:

  * список позиций;
  * контактные данные;
  * способ оплаты;
  * статус и история (опционально).

**Функционал:**

* Требует привязки по `telegram_id`:

  * `GET /orders?telegram_id=...` или отдельный endpoint `/me/orders`.
* Обновление статуса в реальном времени (опционально, через polling или WebSocket; в минимальном варианте — ручное обновление по pull-to-refresh).

### 4.3. BLoC и модули Mini App

Рекомендуемые BLoC-и:

* `AuthBloc` — состояние авторизации/валидированного пользователя.
* `ConfigBloc` — бизнес-конфиг (цвета, логотип, настройки оплат).
* `CatalogBloc` — категории, список продуктов, фильтрация, поиск.
* `ProductBloc` — детальная карточка товара.
* `CartBloc` — корзина, работа с hive.
* `CheckoutBloc` — оформление заказа, отправка на backend, управление шагами оплаты.
* `OrdersBloc` — список заказов, детализация.
* Опционально — `AppBloc` для глобальных состояний (тема, i18n, network status).

Каждый BLoC:

* чётко определённые события и состояния;
* покрывается unit-/bloc-тестами на основную бизнес-логику.

### 4.4. Offline и кеширование в Mini App

* **Cart**:

  * хранится в `hive` локально;
  * не зависит от сети;
  * при восстановлении сети и оформлении заказа корзина отправляется на сервер.
* **Каталог**:

  * кеш последней успешной загрузки на диск;
  * TTL ~30 секунд (как минимум задан в конфиге);
  * при отсутствии сети:

    * показывать кешированный каталог (с пометкой «Нет подключения»);
    * позволять формировать корзину;
    * при попытке оформить заказ без сети — показывать понятную ошибку.
* Проверка сети:

  * использование `connectivity_plus` или аналогичного пакета.

### 4.5. Безопасный платёжный поток

* Все параметры оплаты (сумма, валюты, идентификатор заказа) — **только из backend**.
* Клиент **не хранит и не изменяет** критичные поля оплаты.
* Оплата всегда инициируется через:

  * `checkout_url` от backend;
  * либо API Telegram (если провайдер — Telegram Payments).
* Статус заказа определяется **только ответом backend** (по webhook от провайдера).

## 5. Admin Web Panel (Flutter Web)

Admin Panel так же реализуется на Flutter Web — отдельный entrypoint и роутинг.

### 5.1. Общие требования

* Доступ с десктопных и мобильных браузеров (адаптивная вёрстка).
* Авторизация:

  * Login по email/password (или username/password) через `POST /auth/login`.
  * Получение access/refresh tokens (если предусмотрено backend’ом).
  * Logout, автообновление токенов (refresh flow).
* Роли:

  * минимум: `owner`, `staff`.
  * UI/доступ к функционалу привязан к роли (например, настройки бизнеса могут быть доступны только owner).

### 5.2. Основные разделы Admin Panel

Рекомендуемый набор экранов/страниц:

1. **Login**

   * форма входа;
   * обработка ошибок (неверный логин/пароль, блокировка, и т.п.).

2. **Dashboard (опционально)**

   * краткая статистика: количество активных заказов, заказы за сегодня, выручка за период и т.п.

3. **Продукты (Products)**

   * Список товаров:

     * таблица: название, категория, цена, статус (активен/скрыт), наличие;
     * сортировка, поиск по названию, фильтр по категории.
   * Создание/редактирование товара:

     * форма: название, категория, цена, описание, изображения (загрузка через backend), статус.
   * Удаление/архивация товара:

     * мягкое удаление или отключение (соответствует backend-модели).

4. **Категории (Categories)**

   * Список категорий:

     * таблица: название, порядок сортировки, статус.
   * CRUD-операции:

     * создание, редактирование, удаление/отключение.

5. **Заказы (Orders)**

   * Список заказов:

     * таблица: дата/время, номер заказа, клиент, сумма, статус, способ оплаты.
     * фильтры по статусу, дате, способу оплаты.
   * Детальный просмотр заказа:

     * список позиций;
     * контакты клиента;
     * комментарий клиента (если есть);
     * история изменения статусов (если поддерживается backend).
   * Управление статусами:

     * действия: `Accept`, `Prepare`, `Ready`, `Complete`, `Cancel` (точное соотнесение с backend-статусами);
     * изменение статуса — через `PATCH /orders/{id}`.
   * Нотификации о новых заказах:

     * polling каждые N секунд (например, 10) — `GET /orders?status=new&since=...`;
     * визуальное уведомление (badge, всплывающий toast);
     * опционально — WebSocket для более оперативного обновления.

6. **Настройки бизнеса (Business Settings)**

   * Основные настройки:

     * название;
     * логотип;
     * контактные данные;
     * рабочее время (если есть в backend).
   * Настройки интеграции:

     * способы оплаты;
     * настройки Telegram (Webhook URL, токены, и т.п.) — если backend предоставляет соответствующий API.
   * Настройки интерфейса:

     * брендовый цвет (используется Mini App);
     * локаль по умолчанию.

### 5.3. UI/UX требований Admin Panel

* **Адаптивная верстка**:

  * боковое меню на десктопе;
  * бургер-меню/Drawer на мобильных.
* Единая **дизайн-система**:

  * общие компоненты: кнопки, поля ввода, карточки, таблицы, модальные окна.
* Использование Flutter widgets:

  * `DataTable` или сторонние таблицы для списков (например, `PaginatedDataTable` / кастомные решения для большего контроля).
* Чёткая визуализация статуса заказов цветами и значками.

### 5.4. Архитектура Admin Panel

* Аналогичная слоистая архитектура с Mini App:

  * `presentation`, `domain`, `data`, `core`.
* Рекомендуемые BLoC-и:

  * `AuthBloc` — управление авторизацией и токенами.
  * `ProductsBloc` / `ProductEditBloc`.
  * `CategoriesBloc` / `CategoryEditBloc`.
  * `OrdersBloc` / `OrderDetailsBloc`.
  * `SettingsBloc`.
* Навигация:

  * Router (Navigator 2.0 или `go_router`) с support deep links и удобными URL для страниц (например, `/orders`, `/orders/:id`, `/products`, `/products/new`).

## 6. Безопасность на фронтенде

* **Хранение токенов:**

  * На Web — минимизация рисков:

    * предпочтительно хранить токены в memory + refresh через backend (по session cookie, если это реализовано сервером);
    * если используется `localStorage`/`sessionStorage` — осознанно, с учётом XSS-рисков.
* **Telegram init_data**:

  * не доверять ни одному полю, кроме как через проверку на backend;
  * фронт передаёт полностью `init_data` как строку.
* **Данные оплат**:

  * цены, суммы — только с backend;
  * любая логика расчёта стоимости при оплате — на сервере;
  * фронт использует итоги, возвращённые сервером.
* **Валидация входных данных на фронте**:

  * базовая (формы, форматы, обязательность полей), но окончательная валидация — на backend.

## 10. Acceptance criteria (Frontend)

### 10.1. Mini App

* Работает внутри Telegram WebApp:

  * успешно получает `init_data` и `business_slug`;
  * отправляет их на `/telegram/validate_init_data` и корректно обрабатывает ответ.
* Каталог:

  * загружается с сервера;
  * поддерживаются категории, поиск, фильтрация;
  * добавление в корзину (включая сохранение в `hive`).
* Корзина:

  * позволяет изменять количество товаров, удалять позиции;
  * корректно считает сумму.
* Checkout:

  * оформление заказа с выбранным способом оплаты;
  * при онлайн-оплате — переход на `checkout_url` и корректный учёт статуса по webhook’у (через polling).
* История заказов:

  * отображает заказы, привязанные к `telegram_id` пользователя;
  * корректно показывает статусы.

### 10.2. Admin Panel

* Авторизация:

  * вход/выход, работа с ролями (`owner`, `staff`).
* CRUD по товарам и категориям:

  * создание, редактирование, отключение/архивация;
  * списки с фильтрацией и поиском.
* Работа с заказами:

  * отображение списка заказов;
  * просмотр деталей;
  * изменение статусов с немедленным обновлением в UI и успешным обновлением на backend.
* Нотификации по заказам:

  * при появлении новых заказов (через polling/WebSocket) отображается уведомление без перезагрузки страницы.
* Настройки бизнеса:

  * изменения сохраняются и корректно отражаются на Mini App (как минимум — брендовый цвет и название).
